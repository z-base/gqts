GQSCD-Core defines how hardware, firmware, operating system, and application components on an end-user device can provide verifiable signature-creation properties aligned to qualified-signature security objectives ([EU-910/2014], [EU-2016/650]).

This specification is technical and does not itself grant legal qualification ([EU-910/2014], [EU-TRUSTED-LISTS]).

This is an Editor's Draft.

AI assistance disclosure: this draft used OpenAI GPT-5 Codex for drafting support, structure proposals, and editorial rewrites. Human editor review determined normative content, accepted/rejected changes, and final wording.

Scope

  • Defines a component security profile for end-user signing devices ([EU-910/2014], [EU-2016/650]).
  • Defines explicit threat assumptions and non-trust assumptions ([ETSI-319401]).
  • Defines a primary Web Profile and an EU Compatibility Profile ([WEBCRYPTO-L2], [WEBAUTHN-L3], [EU-2024/2982]).
  • Defines a mapping from GQSCD properties to eIDAS Annex II style objectives ([EU-910/2014], [EU-2016/650]).

Positioning and Adoption Model

This document is a GQSCD-Core baseline specification: a formal technical base document that defines cross-platform security invariants, evidence artifacts, and clause-level Annex II traceability.

GQSCD-Core is designed to be extended by [=implementing specification=] documents for platform-specific implementation details, test suites, and jurisdiction-specific evidence publication workflows.

  • Manufacturer adoption objective: build certifiable configurations from the outset and prepare auditable hardware/firmware/software evidence for certification and notification pipelines ([EU-910-CONS], [EU-2025/1570]).
  • EU deployment objective: accelerate compliant device availability for the EU market by aligning engineering outputs with Article 30/31 and Annex II evidence expectations ([EU-910-CONS], [EU-ANNEX-II]).
  • Decentralization and digital sovereignty objective: reduce single-point trust dependencies through [=zero-trust verification model=] checks and open interoperability, improving market contestability, individual protection, and cost-effective implementation choice ([NIST-SP-800-207], [RFC7515], [RFC9052], [WEBAUTHN-L3]).
  • Cross-jurisdiction interoperability objective: provide a neutral, web-first technical baseline that other jurisdictions can map into their legislation or assurance governance while separating technical validity from legal recognition ([RFC7515], [RFC9052], [WEBAUTHN-L3]).

The key words MUST, MUST NOT, SHOULD, SHOULD NOT, and MAY are to be interpreted as described in BCP 14 [[RFC2119]] [[RFC8174]].

Requirement drafting follows W3C conformance conventions and ETSI-style security requirement traceability; each security claim in normative and profile matrices includes explicit reference basis links ([ETSI-319401]).

A conforming implementation MUST satisfy all requirements of at least one conformance class in this document.

Terminology

end-user device
A phone, tablet, laptop, desktop, or hardware authenticator operated by a natural person in signature workflows.
signature creation data
Data uniquely used by the signatory to create an electronic signature; aligned with eIDAS terminology for electronic signature creation data ([EU-910/2014]).
signature creation key
Cryptographic private key material used as [=signature creation data=] in digital signature mechanisms ([RFC7515], [RFC9052]).
hardware-backed key
A [=signature creation key=] whose generation, storage, and cryptographic use are enforced by protected hardware boundaries with non-exportability controls ([ANDROID-KEYSTORE], [MSFT-TPM], [APPLE-SE]).
qualified signature creation device (QSCD)
Device class defined by eIDAS and evaluated against Annex II outcomes and referenced assessment standards ([EU-910/2014], [EU-2016/650]).
sole control
Security objective that signature creation data can be used only under control of the legitimate signatory, as required by Annex II outcome language ([EU-910/2014]).
trusted execution environment (TEE)
Isolated execution context used for security-critical operations separate from the rich operating environment ([ANDROID-TRUSTY]).
secure boot chain
Firmware and boot verification sequence that permits execution only of cryptographically verified boot components ([ANDROID-AVB], [MSFT-SECURE-BOOT], [APPLE-BOOT]).
measured boot evidence
Measurement artifacts (for example TPM PCR values and event logs) describing boot state for remote verification ([MSFT-TPM], [SYSTEMD-PCR]).
user intent ceremony
Complete interaction in which the signer can inspect what is being signed and explicitly authorize the operation, with security-critical portions mediated by trusted platform paths when available ([ANDROID-PROTECTED-CONFIRMATION], [WEBAUTHN-L3]).
user verification
Authenticator procedure that validates the user through local authentication mechanisms before sensitive operations are performed ([WEBAUTHN-L3]).
attestation evidence
Cryptographically verifiable evidence about key origin, authenticator/device properties, or execution state used by verifiers during trust decisions ([WEBAUTHN-L3], [ANDROID-ATTESTATION]).
zero-trust verification model
Verification model where each decision is based on explicit per-transaction evidence and policy evaluation, rather than implicit trust in network location, client software, or governance labels alone ([NIST-SP-800-207], [RFC7515], [RFC9052]).
trusted computing base
Set of hardware, firmware, and software components that must behave correctly to uphold stated security properties ([ETSI-319401]).
evidence record
Machine-verifiable artifact set used to substantiate technical or legal property claims ([ETSI-319401], [EU-2016/650]).
web profile
Primary interoperability profile in this specification, based on globally deployed web and internet standards ([WEBCRYPTO-L2], [WEBAUTHN-L3], [RFC7515], [RFC9052]).
eu compatibility profile
Mapping layer from [=web profile=] artifacts to EU legal and ecosystem evidence requirements ([EU-910/2014], [EU-2016/650], [EU-2024/2979], [EU-2024/2982]).
qualified status claim
Claim that an actor, service, device, or signature has legal qualified status within a defined jurisdiction; such claims require authoritative status evidence ([EU-TRUSTED-LISTS], [EU-QSCD-NOTIFICATIONS]).
eidas framework
EU legal framework for electronic identification and trust services established by Regulation (EU) No 910/2014 and its amendments ([EU-910/2014], [EU-2024/1183]).
eudi wallet ecosystem
EU Digital Identity Wallet ecosystem specified by the 2024 eIDAS amendment and implementing acts defining integrity, protocol, and interoperability requirements ([EU-2024/1183], [EU-2024/2979], [EU-2024/2982]).
implementing specification
Companion specification that profiles GQSCD-Core for a specific platform, deployment model, testing method, or jurisdictional evidence workflow while preserving core conformance semantics.

Threat Model

Implementations MUST model attacker capabilities including network control, malicious local process execution, UI overlay/tapjacking, software rollback attempts, and dishonest relying-party behavior ([ETSI-319401], [ANDROID-PROTECTED-CONFIRMATION]).

A conformance report MUST explicitly list the [=trusted computing base=], trust assumptions, and excluded trust assumptions ([ETSI-319401]).

Implementations MUST NOT treat UI labels, software allowlists, or relying-party declarations as sufficient security evidence without cryptographic verification ([EU-910/2014], [RFC7515], [RFC9052]).

Threat Required mitigation Reference basis
T-01 key exfiltration Use [=hardware-backed key=] custody and deny raw private-key export. [ANDROID-KEYSTORE], [MSFT-TPM], [WEBAUTHN-L3]
T-02 UI deception Bind [=user intent ceremony=] to exact payload digest and policy. [ANDROID-PROTECTED-CONFIRMATION], [WEBAUTHN-L3], [APPLE-SECACCESSCONTROL]
T-03 rollback Use anti-rollback with signed update provenance. [ANDROID-AVB], [ANDROID-KEYMINT-VERSION-BINDING], [MSFT-SECURE-BOOT], [APPLE-LOCALPOLICY]
T-04 registry spoofing Validate signed, freshness-checked [=evidence record=] artifacts. [EU-TRUSTED-LISTS], [EU-QSCD-NOTIFICATIONS], [RFC7515], [RFC9052]

This section is informative and does not add conformance requirements beyond the normative sections.

QSCD hardware requirements in [=eidas framework=] are outcome-oriented, not a fixed catalog of chip models. Annex II security outcomes can be summarized as: [=signature creation data=] is unique/confidential, not feasibly derivable/forgeable, protected against unauthorized use, and the signer can inspect what is being signed without silent alteration by the device ([EU-910/2014]).

Commission Implementing Decision (EU) 2016/650 links these outcomes to specific assessment standards and protection profiles. QFC Qualified Firmware Components (Informative)

This section is informative and does not add conformance requirements beyond the normative sections.

This section defines a firmware and boot-chain component taxonomy. Like hardware, firmware requirements in Annex II are outcome-oriented. The objective is to show concrete, commodity firmware components that can enforce authenticated boot, rollback resistance, non-extractable key use constraints, and malware-resistant user-intent confirmation ([EU-910/2014], [EU-2016/650]).

Decision (EU) 2016/650 links these outcomes to specific evaluation methodologies. Therefore, this section is non-normative implementation guidance, not a legal status claim ([EU-2016/650], [CEN-419241-1]).

Smartphones and Tablets (AOSP-Class)

  • Boot ROM and hardware root of trust: immutable first-stage code that anchors chain-of-trust verification ([ANDROID-AVB]).
  • AVB in bootloader (Verified Boot 2.0): signed partition metadata, `vbmeta`, and `libavb` integration for boot-time verification ([ANDROID-AVB], [ANDROID-AVB-README]).
  • Firmware rollback protection: AVB rollback indexes in `vbmeta` plus device-stored last-seen rollback indexes in tamper-evident storage ([ANDROID-AVB-README]).
  • KeyMint trusted application in TEE: version binding and key ratcheting tied to verified boot state and patch levels ([ANDROID-KEYMINT-VERSION-BINDING]).
  • Trusted UI and Protected Confirmation: cryptographic confirmation tokens proving the user approved exact message bytes ([ANDROID-PROTECTED-CONFIRMATION], [ANDROID-CONFIRMATIONPROMPT]).
  • StrongBox KeyMint path (where present): dedicated secure processor firmware for stronger isolation than generic TEE-only flows ([ANDROID-KEYSTORE]).

Apple-Class Firmware Components

  • Boot ROM -> LLB -> system-paired firmware: chain-of-trust boot where LLB verifies signatures and loads paired firmware ([APPLE-BOOT]).
  • LocalPolicy signed by Secure Enclave: boot and runtime policy object signed by SEP-local key material ([APPLE-LOCALPOLICY]).
  • LocalPolicy anti-replay: LLB checks anti-replay state sourced from Secure Enclave-attached secure storage to block replay of older policies ([APPLE-LOCALPOLICY]).

PC and Laptop Firmware Components

  • UEFI Secure Boot policy stores: `PK`, `KEK`, `db`, and `dbx` control what low-level boot components are trusted or revoked ([MSFT-SECURE-BOOT]).
  • Measured boot into TPM PCRs: firmware and boot components are measured and extended into PCRs for later quote-based attestation ([MSFT-TPM], [SYSTEMD-PCR]).
  • PCR7 policy signaling: commonly used to represent Secure Boot policy/state and related key-configuration evidence ([SYSTEMD-PCR]).

Certification Boundary for Firmware Claims

These firmware components are strong enforcement primitives, but legal QSCD qualification depends on certification of the exact evaluated firmware/boot-chain target and related conformity-assessment outcomes ([EU-910-CONS], [EU-2016/650], [CEN-419241-1]).

Firmware teams should treat rollback counters, boot policy measurements, and trusted-confirmation bindings as first-class certification artifacts, because notification requires specific configuration identifiers and report references ([EU-2025/1570]).

  • Android: Android Keystore with KeyMint/Keymaster and optional StrongBox; authentication-bound key parameters; BiometricPrompt; Protected Confirmation and confirmation-bound key use ([ANDROID-KEYSTORE], [ANDROID-BIOMETRICPROMPT], [ANDROID-PROTECTED-CONFIRMATION]).

Interoperability Profiles

Web Profile (Primary)

  • MUST be the baseline profile for interoperability ([WEBCRYPTO-L2], [WEBAUTHN-L3]).
  • MUST support JOSE/JWS/JWK or COSE/CBOR signature containers ([RFC7515], [RFC7517], [RFC9052], [RFC8949]).
  • SHOULD support WebAuthn for hardware authenticator mediation where applicable ([WEBAUTHN-L3]).
  • MAY use DID Core, VC Data Model, Data Integrity, OpenID4VP, and SD-JWT VC when credentials are needed and deployment maturity requirements are satisfied ([DID-CORE], [VC-DM2], [VC-DI], [OPENID4VP], [SD-JWT-VC]).
  • MUST define security properties in physical reality terms: key custody, tamper resistance, and verifiable user intent ([EU-910/2014], [EU-2016/650]).

EU Compatibility Profile (Mapping Layer)

  • MUST map Web Profile artifacts to EU objective/evidence checks ([EU-910/2014], [EU-2016/650]).
  • MUST NOT replace the Web Profile as the primary technical model.
  • Any [=qualified status claim=] MUST carry an [=evidence record=] linked to current trusted-list and/or notification data ([EU-TRUSTED-LISTS], [EU-QSCD-NOTIFICATIONS]).
  • MUST output technical validity and legal recognition as separate results ([ETSI-319401]).

EU Certification and Notification Path (Informative)

In EU operational terms, a QSCD certificate is evidenced by a certificate of conformity and certification report for a specific configured device, and discoverability is achieved through Member State notification and Commission publication/list maintenance.

Article 30 defines conformity certification routes and lifecycle constraints, Article 31 defines Commission notification/publication mechanics, Implementing Decision (EU) 2016/650 defines recognized assessment standards routes, and Implementing Regulation (EU) 2025/1570 defines notification data fields and procedures ([EU-910-CONS], [EU-2016/650], [EU-2025/1570]).

Step ID Certification/notification step Required evidence artifact Legal anchor
EUCERT-01 Define evaluated target and category (user-managed environment or remote/managed environment) Configuration definition with exact hardware/firmware/software identifiers [EU-910-CONS], [EU-2025/1570]
EUCERT-02 Run conformity assessment via designated body using Article 30 route Certification report and certificate of conformity aligned to Annex II objectives [EU-910-CONS], [EU-2016/650], [EU-ANNEX-II]
EUCERT-03 Apply lifecycle maintenance controls after issuance Validity period control (max 5 years) and vulnerability-assessment records every 2 years [EU-910-CONS]
EUCERT-04 Submit Member State notification package to Commission Notification payload including report URL, status dates, cert body identity, and version-specific configuration details [EU-910-CONS], [EU-2025/1570]
EUCERT-05 Commission publication/list maintenance and status updates Published list entry and change records for certified/no-longer-certified status [EU-910-CONS], [EU-QSCD-NOTIFICATIONS]
qscd certification evidence tuple
{ certificate of conformity, certification report, certification report URL, certified configuration identifiers (hardware/firmware/software versions), validity dates and status, certification body identity }

The tuple above is the minimum implementation-facing artifact set this specification expects verifiers and ecosystem operators to ingest when evaluating EU legal qualification claims ([EU-2025/1570], [EU-910-CONS]).

ENISA guidance may be used to reduce interpretation ambiguity in evaluation/certification planning, but does not replace legal requirements or designated-body outcomes ([ENISA-QSCD-EUCC], [EU-910-CONS]).

Conformance Classes

Class A: Device Stack

  • MUST satisfy all normative requirements in this specification for device-side behavior.
  • MUST implement the component matrix with machine-verifiable evidence artifacts ([ETSI-319401], [EU-2016/650]).
  • MUST implement signed update verification and anti-rollback ([ANDROID-AVB], [MSFT-SECURE-BOOT], [APPLE-BOOT]).

Class B: Verifier

  • MUST validate cryptographic signatures and policy constraints ([RFC7515], [RFC9052]).
  • MUST process [=evidence record=] artifacts and emit explicit pass/fail reasons ([ETSI-319401]).
  • MUST emit clause-level Annex II mapping results using [=ANNEXII-1a=], [=ANNEXII-1b=], [=ANNEXII-1c=], [=ANNEXII-1d=], and [=ANNEXII-2=].
  • MUST separate technical validity from legal recognition outcomes ([EU-910/2014]).

Class C: EU Compatibility Verifier

  • MUST satisfy Class B.
  • MUST verify trusted-list and QSCD-notification evidence for legal qualification claims ([EU-TRUSTED-LISTS], [EU-QSCD-NOTIFICATIONS]).
  • MUST NOT infer legal qualification from cryptographic success alone ([EU-910/2014], [EU-2016/650]).

Security Considerations

  • Approved software lists and governance labels are not cryptographic controls by themselves ([RFC7515], [RFC9052], [EU-910/2014]).
  • Implementations SHOULD apply a [=zero-trust verification model=] so that every signature decision is derived from fresh, signed evidence and explicit policy checks ([NIST-SP-800-207], [RFC7515], [RFC9052]).
  • Backup and recovery flows MUST preserve both confidentiality and auditability ([ETSI-319401], [ETSI-319411-2]).
  • Signer authorization channels MUST defend against replay, overlay, and confused-deputy attacks ([ANDROID-PROTECTED-CONFIRMATION], [WEBAUTHN-L3], [RFC7515]).

Privacy Considerations

  • Implementations SHOULD minimize personally identifying data in evidence artifacts ([VC-DM2], [VC-DI], [EU-2024/2979]).
  • Public verification endpoints MUST NOT disclose private sensitive material ([RFC9110], [ETSI-319401]).
  • Selective disclosure SHOULD be used where policy permits ([SD-JWT-VC], [VC-DM2]).
